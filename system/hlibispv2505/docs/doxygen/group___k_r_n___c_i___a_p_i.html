typedef struct KRN_CI_PIPELINE KRN_CI_PIPELINE
Adding a new HW module
In order to add a new HW to be configure you need to:
  • define a new CI_MODULE_MOD structure in ci_modules_structs.h and its associated functions in ci_modules.h (if it needs special setup)
  • add this structure to CI_PIPELINE
  • update the CI_MODULE_UPDATE flags
  • update INT_CI_PipelineUpdate() to take into account your new flag and the module
  • add a new HW_CI_Load_MOD (and HW_CI_Reg_MOD if needed) into ci_hwstruct.h
  • update KRN_CI_PipelineUpdate() to use your new functions (be aware that register access are limited)
  • update the unit tests CI_Config::update() to test the independant update of your new structure

enum KRN_CI_SHOT_eSTATUS
Enumerator
CI_SHOT_AVAILABLE 

KRN_CI_SHOT is ready to be given to HW for processing

The status changes to CI_SHOT_PENDING.

CI_SHOT_PENDING 

KRN_CI_SHOT is currently used by HW (or waiting to be used by the HW).

When HW finished processing the status changes to CI_SHOT_PROCESSED.

If capture failed the buffer is returned as CI_SHOT_AVAILABLE

CI_SHOT_PROCESSED 

KRN_CI_SHOT has been processed by HW but not acquired by user-space yet

When user acquires it the status changes to CI_SHOT_SENT.

CI_SHOT_SENT 

KRN_CI_SHOT is acquired by user-space and cannot be used.

When user releases it the KRN_CI_SHOT is moved as CI_SHOT_AVAILABLE.

IMG_UINT32 KRN_CI_MMUTilingAlignment ( IMG_UINT32  ui32TilingStride)

Computes the virtual address alignment needed according to a tiling stride.

HW needs the 1st address tiled memory to be alignment to a certain multiple to avoid going backward when accessing it.

It is defined in MMU spec as 1<<(x_tile_stride+8+5+s), x_tile_stride being the register value and s being 0 for 256x16 scheme and 1 for 512x8 scheme.

IMG_RESULT KRN_CI_PipelineStartCapture ( KRN_CI_PIPELINE *  pPipeline)

Starts the capture.

In that order:

  • if using DE point, try to enable the needed point
  • try to acquire the SW context
  • checks the size of the LSH grid
  • update the configuration
  • try to acquire the needed DPF internal buffer
  • map the element of available and sent list to the MMU
  • map the availableBuffer elements to the MMU
  • map the LSH grid to the MMU
  • map the DPF read map to the MMU
  • set the tiling stride
  • flush the MMU cache
  • start the HW

this is done in the lock because it only 1 capture per context is possible and only 1 file per context is used

IMG_RESULT KRN_CI_PipelineStopCapture ( KRN_CI_PIPELINE *  pPipeline)

Stops the capture.

In that order:

  • stops the HW
  • unmap all memory from available and sent list
  • clear all elements from pending list back to available list (and unmap from MMU)
  • clear all elements from processed list back to available list (and unmap from the MMU)
  • unmap all imported buffer from avaiableBuffer list from MMU
  • unmap the DPF read map from MMU
  • unmap the LSH grid from MMU
  • release the DPF internal buffer
  • release the SW context

Updates the pMatrixUsed->sBuffer.used counter. This counter is also updated in KRN_CI_PipelineUpdateMatrix() (when starting the capture or changing the matrix while the capture is runnign)

Warning
As interrupts are stopped before cleaning the pending list it is not needed to lock the spinlock
